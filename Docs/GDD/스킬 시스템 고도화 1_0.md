# 🚀 [GDD] 확장형 모듈식 스킬 시스템 (v1.0)

## 1. 시스템 개요
기존의 단일 구조(Monolithic) 스킬 데이터를 **모듈형(Component-based)**으로 개편한다. 하나의 스킬이 여러 개의 독립적인 '행동(Action)'을 순차적으로 실행할 수 있도록 배열 형태로 확장하며, 각 행동마다 정교한 '타겟팅(Targeting)' 옵션을 부여하여 코드 수정 없이 데이터만으로 복잡한 기믹(예: 딜탱 복합기, 조건부 힐링)을 구현하는 것을 목표로 한다.

---

## 2. 데이터 스키마 구조 (`SkillData.json`)

### 2.1 스킬 루트 (Skill Root)
스킬의 기본 정보와 시각적 연출, 그리고 실행할 행동들의 리스트를 담는다.
* **Id (`uint`)**: 스킬 고유 식별자.
* **DisplayName (`string`)**: 스킬 이름.
* **EffectPrefabPath (`string`)**: 스킬 발동 시 출력할 메인 이펙트 리소스 경로.
* **Actions (`Array`)**: [핵심] 이 스킬이 실행할 순차적 행동들의 리스트.

### 2.2 행동 객체 (Action Object)
배열 안에 들어가는 개별 단위 논리 로직.
* **ActionType (`string`)**: 행동의 본질 (`Attack`, `Heal`, `Shield`, `Buff`).
* **Target (`Object`)**: 해당 행동이 적용될 대상을 찾는 규칙 객체.
* **BaseMultiplier (`float`)**: 주 스탯(공격력 등)에 곱해질 계수.
* **StatusEffects (`Array`)**: 행동 적중 시 타겟에게 부여할 부가 효과(상태 이상/버프) 리스트.

### 2.3 타겟 객체 (Target Object)
타겟을 탐색하고 확정하는 규칙 명세.
* **Team (`string`)**: 피아 식별 (`Ally`: 아군 파티, `Enemy`: 적 몬스터 웨이브).
* **Strategy (`string`)**: 타겟 탐색 알고리즘 (아래 3. 타겟팅 알고리즘 명세 참고).
* **MaxCount (`int`)**: 타겟팅할 최대 인원수 (0이면 `Strategy`에 따라 자동 결정 또는 전체).
* **TargetIndex (`int`)**: (선택) `Strategy`가 `FixedIndex`일 경우에만 참조하는 특정 슬롯 번호(0~4).

## 3. 타겟팅 알고리즘 명세 (Strategy Enum)
`Target.Strategy`에 입력될 수 있는 값의 정의. 클라이언트의 `TargetingSystem`은 이 값을 기반으로 대상을 색인한다.

* **`Front`**: 최전방(Index 0에 가까운) 살아있는 대상부터 탐색. (일반적인 공격)
* **`Back`**: 최후방(Index 끝에 가까운) 살아있는 대상부터 탐색. (암살자 기믹)
* **`LowestHP`**: 현재 HP 비율(%)이 가장 낮은 살아있는 대상. (스마트 힐링)
* **`HighestAtk`**: 공격력이 가장 높은 대상. (핵심 딜러 저격용)
* **`Self`**: 스킬을 시전한 주체 본인.
* **`FixedIndex`**: 지정된 `TargetIndex` 위치에 있는 대상. (죽었으면 무효 처리)
* **`All`**: 조건에 맞는 살아있는 전체 대상.
* **`Random`**: 살아있는 대상 중 무작위.

## 4. 데이터 활용 예시 (JSON)

### 예시 A: 복합 지원 스킬 (최전방 탱커 지정 힐 + 자신 방어력 증가)
```json
{
  "Id": 6001,
  "DisplayName": "성녀의 가호",
  "EffectPrefabPath": "Effects/Skills/HolyAegis",
  "Actions": [
    {
      "ActionType": "Heal",
      "Target": {
        "Team": "Ally",
        "Strategy": "FixedIndex",
        "TargetIndex": 0,
        "MaxCount": 1
      },
      "BaseMultiplier": 2.5,
      "StatusEffects": []
    },
    {
      "ActionType": "Buff",
      "Target": {
        "Team": "Ally",
        "Strategy": "Self",
        "MaxCount": 1
      },
      "BaseMultiplier": 0.0,
      "StatusEffects": [
        { "EffectType": "DefUp", "Value": 0.5, "Duration": 10.0, "Probability": 1.0 }
      ]
    }
  ]
}
```

### 예시 B: 공방 일체 (적 전방 2명 공격 + 아군 딸피 1명 힐)
```json
{
  "Id": 6002,
  "DisplayName": "흡혈의 일격",
  "EffectPrefabPath": "Effects/Skills/VampiricStrike",
  "Actions": [
    {
      "ActionType": "Attack",
      "Target": {
        "Team": "Enemy",
        "Strategy": "Front",
        "MaxCount": 2
      },
      "BaseMultiplier": 2.0,
      "StatusEffects": []
    },
    {
      "ActionType": "Heal",
      "Target": {
        "Team": "Ally",
        "Strategy": "LowestHP",
        "MaxCount": 1
      },
      "BaseMultiplier": 1.5,
      "StatusEffects": []
    }
  ]
}
```

## 5. 시스템 확장 시 고려 사항 (Tech Note)
1.  **실행 순서 보장:** `Actions` 배열에 정의된 순서(인덱스 0부터)대로 로직이 동기적으로 실행되어야 한다. (예: 적을 먼저 때려서 죽인 후, 아군을 회복시키는 순서).
2.  **타겟 무효화 처리:** 스킬 시전 시점과 Action 실행 시점 사이에 타겟이 사망할 수 있으므로, 각 Action을 실행하기 직전에 `TargetingSystem`을 통해 유효성을 한 번 더 검증해야 한다.
